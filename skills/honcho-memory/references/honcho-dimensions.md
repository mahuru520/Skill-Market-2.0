# Honcho 六大维度详解

> 读取时机：需要更新画像、或在复杂交互后做画像推断时。
> 不需要每次对话都读，按需加载。

---

## 维度一：技术水平（Skill Level）

**范围**：`0.0`（新手）→ `1.0`（专家）

**推断信号**：

| 信号 | → 低水平 | → 高水平 |
|------|---------|---------|
| 提问方式 | "怎么弄"、"那个..." | 使用专业术语、问原理 |
| 错误处理 | 照抄答案不懂原因 | 问"为什么"、想改源头 |
| 操作速度 | 慢，逐一确认 | 快，直接给出意图 |
| 文件访问 | 需要告知路径在哪 | 自己找得到 |

**辩证点**：同一用户在不同子领域的水平差异极大。
- Python 专家 ≠ TypeScript 专家
- 调试老手 ≠ 架构设计老手
- → 用 `subDimensions` 替代单一分数

**存储格式**：

```json
{
  "skill": {
    "overall": 0.75,
    "subDimensions": {
      "typescript": 0.85,
      "python": 0.80,
      "llm-agent": 0.70,
      "windows-cli": 0.75
    },
    "confidence": 0.8,
    "lastUpdated": "2026-05-09",
    "evidence": [
      "改 node_modules/@tobilu/qmd/dist/store.js 自定义 embedding 路由",
      "写 qmd-launcher.js XDG 路径注入",
      "理解 LlamaCpp session 机制"
    ]
  }
}
```

## 维度二：交互节奏（Pacing）

**范围**：`0.0`（缓慢细致）→ `1.0`（极速多任务）

**推断信号**：

| 信号 | → 慢节奏 | → 快节奏 |
|------|---------|---------|
| 查询长度 | 长句、详细描述 | 短句、关键词 |
| 等待行为 | 等待解释完成 | 直接"继续"、"下一个" |
| 反馈速度 | 逐条确认后继续 | 直接给出指令序列 |
| 任务类型 | 调试、写文档 | 状态查询、列清单 |

**辩证点**：用户在不同任务上节奏截然相反。
- 写代码快，调试慢
- 简单查询快，架构设计慢
- → 用 `taskPacing` 记录任务类型相关的节奏

**存储格式**：

```json
{
  "pace": {
    "overall": 0.65,
    "taskPacing": {
      "query": 0.85,
      "coding": 0.75,
      "debugging": 0.50,
      "learning": 0.45
    },
    "confidence": 0.75,
    "lastUpdated": "2026-05-09"
  }
}
```

## 维度三：沟通风格（Style）

**谱系**：`formal` ↔ `casual`，`concise` ↔ `verbose`，`humor` ↔ `serious`

**推断信号**：

| 信号 | 风格倾向 |
|------|---------|
| 句式长度 | 短 → 简洁；长 → 详尽 |
| 标点/表情 | 无 emoji → 技术向；有 → 随意 |
| 寒暄 | 无 → 直接；有 → 社交型 |
| 用词 | "请"/"能否" → 正式；"搞"/"跑" → 随意 |

**存储格式**：

```json
{
  "style": {
    "formal": 0.3,
    "concise": 0.75,
    "humor": 0.4,
    "technicalTerms": 0.8,
    "emojiUsage": "rare",
    "confidence": 0.85,
    "lastUpdated": "2026-05-09"
  }
}
```

**注入策略**：

- `formal < 0.4`：回复不端架子，可用缩写和口语
- `concise > 0.7`：减少解释篇幅，用"先结论后细节"策略
- `technicalTerms > 0.7`：直接用专业术语，不用过多类比

## 维度四：目标取向（Goal Orientation）

**范围**：`0.0`（探索型）→ `1.0`（完成型）

| 信号 | 探索型 | 完成型 |
|------|-------|-------|
| 问法 | "还有什么可能？"、"不同方式？" | "怎么实现？"、"给我最终答案" |
| 反馈 | 对扩展建议有兴趣 | "先跑通再说" |
| 文档 | 问"原理" | 问"怎么配置" |

**辩证点**：用户在一次会话中可能先探索后完成。记录 `trajectory`（会话轨迹）。

**存储格式**：

```json
{
  "goal": {
    "overall": 0.6,
    "trajectory": [
      { "turn": 1, "mode": "explore" },
      { "turn": 3, "mode": "complete" }
    ],
    "confidence": 0.7,
    "lastUpdated": "2026-05-09"
  }
}
```

## 维度五：情绪基调（Emotional Tone）

**指标**：valence（正面/负面）、arousal（激活度）、dominantReaction（压力下的典型反应）

**推断信号**：

| 信号 | 含义 |
|------|------|
| "不对"、"错了" | 可能不满，但也是反馈 |
| 反复重问同一件事 | 困惑/不满足 |
| 直接动手调试（不抱怨） | 情绪稳定 |
| 说"烦了"、"算了" | 压力高 |
| 主动问新话题 | 情绪正面，愿意探索 |

**存储格式**：

```json
{
  "emotion": {
    "valence": 0.65,
    "arousal": 0.50,
    "dominantReaction": "focused-debug",
    "stressSignals": [],
    "confidence": 0.6,
    "lastUpdated": "2026-05-09"
  }
}
```

## 维度六：矛盾偏好（Contradictory Preference）⭐

**核心**：矛盾不是要消除，而是显式建模并在响应时兼顾。

**常见矛盾对**：

| 矛盾对 | 典型场景 |
|--------|---------|
| (要示例代码, 讨厌长示例) | 代码任务 |
| (快速出结果, 理解底层原理) | 调试/实现 |
| (要自动, 又要手动控制) | 工具使用 |
| (详细解释, 不要废话) | 教学/教学 |

**矛盾强度**：`0.0`（无矛盾）→ `1.0`（强烈矛盾）

**存储格式**：

```json
{
  "contradictions": [
    {
      "a": "快速出结果",
      "b": "理解底层原理",
      "strength": 0.70,
      "detectedFrom": "QMD embed 调试：既想快跑通，又花时间读 store.js 整个结构",
      "lastUpdated": "2026-05-09"
    }
  ]
}
```

**响应策略**（矛盾存在时）：

- `(要示例, 讨厌长)` → **折叠式**：先给关键片段 + "展开完整代码"
- `(快速出结果, 想理解原理)` → **分层式**：先可用的，再"要深挖哪部分？"
- `(自动, 手动控制)` → **可配置式**：默认自动 + "可手动覆盖"
