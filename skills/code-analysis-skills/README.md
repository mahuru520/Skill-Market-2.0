# 一个 Git 历史自查报告生成器（不是绩效工具）

> 你有没有想过：**自己**的提交节奏是什么样子？哪些时段最容易写出来又删掉的代码？哪些项目你其实很少回头看？

这个项目想做的事情**很有限**：把你**自己**仓库里的 Git 历史，整理成一份带上下文的自查报告，帮你和你的团队（在每个人都同意的前提下）做一次温和的复盘。

它**不是**绩效工具，**不是**摸鱼检测器，**不是**给老板/HR 用来对比同事的工具。这一点这次特别想说清楚。

## ⚠️ 使用前请先读完这一段

这个工具会从 Git 提交历史里提取**单个开发者的活动数据**（提交时间、频率、行为模式等）。在使用前，请明确以下几条：

- ✅ **完全本地运行**，不会把任何数据发到外部服务器
- ✅ 仅适合两类场景：(1) 开发者对**自己**的仓库做自查；(2) 团队在**每位成员都明确同意**的前提下进行复盘
- 🚫 **不应**用于绩效考核、薪酬 / 晋升 / 处分决策
- 🚫 **不应**用于对个人的排名、打分或公开比较
- 🚫 **不应**用于监视员工或未同意的贡献者
- 🔒 生成的报告含个人信息，请妥善保管，不应公开分享
- 📋 使用前请确认符合所在组织的 HR 政策和所在地的隐私 / 劳动法规（如 GDPR）

CLI 与 Skill 入口都设置了硬性门槛：**必须**显式确认（CLI 加 `--i-have-consent`，Skill 设置 `acknowledge_usage_policy: true`），**没有**环境变量绕过。

此外，默认为 **self-scope** 模式：只分析当前本地 Git 用户自己的提交。分析他人需要额外传入 `--multi-author-team-retro` 与逐个 `--consented-author NAME`，**且任何输出都不产生排行榜、多人横向对比表、综合评分或字母等级**。

## 用法很简单

作为 Skill 插件，请使用**意图明确**的请求来触发，例如：

```
"对我自己的仓库做一份 Git 历史自查报告"
"团队复盘（已经征得每个人同意），帮我跑一下 Git 历史汇总"
"我想看看自己最近的提交节奏，做一次自我反思"
```

模糊触发词（"代码分析"、"研发效率"、"开发者评估"等）会被 Skill 主动询问澄清意图，而不是直接对人下结论。

## 它做什么 / 不做什么

这个工具会扫描你指定的（你拥有 / 已获同意的）Git 仓库，按开发者聚合一些**描述性统计**，最终生成报告。它做的事情**只是描述性的**：

- **提交习惯** — 你一天提交几次？每次改多少行？commit message 写了几个字？
- **提交时段** — 提交是怎么分布在一天 / 一周里的？周末 / 深夜的比例？
- **变更模式** — 写了又删的比例（churn）？同一个文件 7 天内重复修改的比例（rework）？
- **代码风格** — 主要写什么语言？commit message 是不是按 Conventional Commits 规范？
- **代码质量痕迹** — 修 bug 的提交占比？回滚比例？大提交比例？测试文件改动比例？

它**不会**告诉你：

- 谁更努力，谁在摸鱼
- 谁应该被绩效改进
- 哪些同事"产出低"

因为这些问题的答案 Git 历史**根本回答不了**。代码评审、设计、辅导、值班、运维，这些重要的工作在 Git 历史里**完全看不到**。一个仅用 Git 数据回答这些问题的工具是**有偏见**的，会**伤人**。

## 几个常见的、被误读的信号

### 深夜提交不等于"勤奋"或"过劳"

```
Peak Hour: 22:00
Late Night Ratio: 100.0%
```

这只能说明这位作者的**提交时间戳**集中在深夜。可能是个人偏好的工作时段、可能是不同时区、可能是把白天的工作批量推送 / squash 后才打了 commit、也可能是值班修线上事故。**仅凭 Git 时间戳没法判断**这个人是熬夜还是按时下班。任何针对个人的结论都需要先和当事人聊一聊。

### Churn 高不等于"代码不稳定"

```
Churn Rate: 66%
```

写完又删除三分之二的代码，常见解释是：探索性原型、重构整理、需求变化、code review 反馈很多。这并不一定是问题。把它当作"和作者聊一聊设计思路"的提示，比当作扣分项有用得多。

### 单次提交两万行也未必"刷量"

```
Avg Lines per Commit: 21,056
Large Commit Ratio: 50.0%
```

最常见的解释是：项目初始化、依赖引入、生成代码 vendor、批量迁移。这种提交对仓库**有用**，但对 review 不友好。可以作为讨论"未来类似变更要不要拆小一点"的契机，但**不是**衡量这个人能力的尺度。

### 只有一个提交不等于"贡献少"

```
Total Commits: 1
```

开源世界里有很多用户的第一个 PR 修了一个关键 bug、补了一段缺失的文档；新人花一周读懂代码再动手写出第一个提交，也是常见且健康的入项节奏。**贡献的形式不止 commit 数量**。

## 技术解剖：架构

```
路径输入 → Scanner 发现仓库 → Analyzer 遍历 Git 历史 → Reporter 生成报告（带使用提示）
```

* **Scanner** 扫描指定路径下的 Git 仓库（最深 5 层），用 GitPython 验证合法性，跳过裸仓库与符号链接循环
* **Analyzer** 用 [PyDriller](https://github.com/ishepard/pydriller) 遍历提交历史，按作者聚合**描述性统计**
* **Reporter** 提供 Markdown / JSON / HTML / PDF 四种输出，**每种输出顶部都会强制带使用提示**

### 几个常被问到的指标定义

**1. 返工率 Rework Ratio**

如果你在 7 天内对同一个文件改了两次以上，就算一次返工。常见原因是 review 反馈、需求迭代、共享所有权的工作 —— 不一定是问题。

**2. 文件所有权 Ownership**

某个文件超过 50% 的修改来自同一作者，则记为该作者"拥有"该文件。这是个**仓库级风险**指标（用于判断 bus factor），**不是**个人评分。

**3. Bus Factor**

某个文件如果只有一个独立贡献者，团队的知识就过于集中。统计每个文件的独立贡献者数取平均值。这个指标关心的是**仓库**的健康度，**不是**某个人的"价值"。

**4. 节奏稀疏度（Cadence-Sparsity）**

只描述提交活动在时间上**有多稀疏**。它**不等于**"摸鱼"或"低投入"。架构师、Reviewer、值班工程师、休假 / 病假人员，都会自然产生稀疏的提交节奏。

## 也可以直接用命令行使用：

```bash
# 默认 self-scope：只分析当前 Git 用户自己的提交
python -m src.main --i-have-consent -r /path/to/repo

# 指定时间范围，生成 HTML 报告
python -m src.main --i-have-consent -r /path/to/repo -s 2025-01-01 -u 2025-12-31 -f html -o report.html

# 同时生成 Markdown + HTML + PDF
python -m src.main --i-have-consent -r /path/to/repo -f "markdown,html,pdf" -o report

# 已获全员同意的团队复盘（必须逐人列出）
python -m src.main --i-have-consent -r /path/to/repo \
  --multi-author-team-retro \
  --consented-author "alice@example.com" \
  --consented-author "bob@example.com"
```

> 不带 `--i-have-consent` 时工具只会打印使用提示并退出；不带 `--multi-author-team-retro` + `--consented-author` 时，只会分析本地 Git 用户自己。这是有意为之。

报告中不会出现多人横向对比表、排行榜、综合评分或字母等级。报告是以**人为单位**的文本叙述 + 各分量分项表，并且每个报告顶部都有使用政策提示。

每个数字背后都有一个真实的人和真实的语境。**指标只是讨论的起点**。

## 写在最后：数据不能定义一个人

很多 Git 分析工具都有一种诱惑 —— 把数据包装成评分、等级、排行榜，让结论看起来很"客观"。但 Git 历史只是工作的一部分（而且是有偏的那一部分），把它包装成对人的判断，会变成对人的伤害。

那个凌晨三点提交代码的同事，也许比任何人都更热爱这个项目；那个 commit message 只写 "fix" 的人，也许正在争分夺秒地修复一个线上事故；那个只贡献了一个提交的人，也许是花了整整一周才理解了代码才敢动手。

写这个工具的初心，是希望它能帮一个人**自己**回头看自己的工作节奏；或者帮一支团队在**每个人都同意**的前提下做一次温和的复盘。**不是**用来扒别人，**不是**用来排名，**不是**绩效工具。

如果你想用它做这些事 —— 这个工具会拒绝运行。这不是 Bug，是设计。

## 🆕 v1.0.7 变更摘要

- 🛡️ **强制同意门槛（无环境变量绕过）**：CLI 必须传 `--i-have-consent`，Skill 必须传 `acknowledge_usage_policy: true`，没有环境变量起 fallback。
- 🔒 **默认 self-scope**：默认只分析当前本地 Git 用户。分析他人需一起传 `--multi-author-team-retro` + 逐个 `--consented-author NAME`。
- 🚫 **移除综合评分 / 字母等级 / 定论句**：报告不再产出 0–100 总分、S/A/B/C/D/E/F 字母段、类似 "top-tier contributor" 的一句话定论。
- 🚫 **移除多人横向对比 / 排行榜**：哪怕是同意过的多人复盘模式，报告都不生成跨作者对比表。
- 🪞 **报告重定位为叙述式自查**：原“开发者评估 / 摸鱼指数”重命名为“反思叙述 / 节奏密度信号”，去除贬损与伪科学表达。
- ⚠️ **每份输出强制使用提示**：Markdown / HTML / PDF 报告顶部都会带使用政策提示。
- 📄 **PDF 输出**：支持生成带样式的 PDF 报告（基于 reportlab）。
- 🔀 **多格式同时输出**：一条命令同时生成 Markdown + HTML + PDF。
