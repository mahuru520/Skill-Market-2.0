# 书骨 · BookBone

> 「一本书读完就忘了？书骨帮你抽走它的骨架，让AI替你记住。」

## 核心理念

书骨不是「把书变成另一份摘要」，是**抽出一本书的认知骨架**——去掉血肉（案例、故事、修辞），留下可运行的结构。

好的一本书 Skill 是一套可操作的思维工具箱：
- 作者用什么**心智模型**看世界？（镜片）
- 作者提出了什么**方法论/检查清单**？（操作步骤）
- 作者创造了什么**关键概念**？（词汇表）
- 这个框架**适用和不适用**什么场景？（边界）

**关键区分**：捕捉的是HOW the book thinks，不是WHAT the book says。

---

## 执行流程

### Phase 0: 确认书目

收到用户输入后，确认以下信息：

| 确认项 | 说明 |
|--------|------|
| **书名+作者** | 确保理解正确，避免同名书籍混淆 |
| **版次** | 第二版和第一版差异可能很大，必须确认 |
| **输入形式** | 用户提供PDF/EPUB / 读书笔记 / 只有书名 |
| **蒸馏目标** | 框架？方法论？案例？思维方式？ |
| **新建 or 更新** | 检查 `.claude/skills/` 是否已有该书 Skill |

**输入形式判断**：

| 形式 | 触发条件 | 策略 |
|------|---------|------|
| **本地原文** | 用户提供PDF/EPUB/全文 | Phase 1直接读取原文，最准确 |
| **笔记模式** | 用户提供读书笔记/摘要 | Phase 1读取笔记 + 网络搜索补充 |
| **搜索模式** | 只有书名，无其他素材 | Phase 1通过网络搜索获取核心内容 |

确认后 → Phase 0.5 创建目录。

### Phase 0.5: 创建Skill目录

```
.claude/skills/[book-slug]/
├── SKILL.md                          # 最终产物
└── references/
    └── book-analysis.md              # 阅读分析记录
```

**完成检查**：
- [ ] 目录已创建
- [ ] book-analysis.md 已创建（空文件，等待Phase 1填充）
- [ ] 如果是更新模式：已读取现有SKILL.md，标注哪些信息需要刷新

**关键规则**：所有分析文件必须存在skill目录内部，Skill必须是自包含的。

---

### Phase 1: 书籍阅读/分析

根据输入形式选择策略：

**本地原文模式**：
1. 读取全书（PDF用pdf skill，EPUB用文本读取）
2. 生成结构化分析记录，写入 `references/book-analysis.md`

**笔记模式**：
1. 读取用户提供的笔记
2. 识别信息缺口（哪些章节/概念缺失）
3. 定向搜索补充

**搜索模式**：
1. 搜索书籍核心章节摘要、书评、作者访谈
2. 尽可能获取目录和关键章节内容
3. 在诚实边界中标注「基于二手来源」

**分析维度**（写入 book-analysis.md 的结构）：

```markdown
# [书名] · 阅读分析

## 书籍概览
- 书名、作者、版次、出版年
- 核心问题：这本书试图回答什么？
- 核心答案：作者的主要论点（200字以内）
- 目标读者：谁应该读这本书？

## 章节摘要
- 逐章/逐部分提取核心论点（每章3-5行）
- 标注「高信息密度章节」

## 关键概念/术语
- 作者自创或重新定义的术语
- 每个术语的定义 + 出现章节

## 方法论/框架
- 书中明确提出的方法论、检查清单、决策树
- 每个框架的输入、过程、输出

## 核心案例
- 书中反复引用的关键案例（3-5个）
- 每个案例说明了什么框架

## 作者的局限与边界
- 作者明确承认的局限
- 本书未覆盖的领域
- 可能的批评角度

## 与其他书的关系
- 作者引用了哪些重要文献/思想
- 本书在同类书籍中的位置
```

**Phase 1 完成检查**：
- [ ] 所有核心章节都有摘要
- [ ] 关键概念列表完整
- [ ] 方法论/框架已提取
- [ ] 高信息密度章节已标注

展示分析摘要给用户确认 → 进入 Phase 2。

---

### Phase 2: 框架提取

读取 `references/book-analysis.md`，执行结构化提炼。

先读取 `references/extraction-methodology.md` 获取三重验证方法论。

#### 2.1 心智模型提取（3-7个）

**三重验证**（详见 extraction-methodology.md）：
- **章间复现**：同一框架在书中至少2个不同章节/场景中出现
- **生成力**：能推断作者对书中未讨论问题的立场
- **排他性**：不是同类书的标配视角

操作步骤：
1. 扫描 book-analysis.md，列出所有候选论点（通常10-20个）
2. 对每个候选执行三重验证
3. 按排他性强度排序，取top 3-7个
4. 记录格式：名称、一句话、原文定义、证据（引用章节）、应用方式、局限性

#### 2.2 方法论/检查清单提取（3-8条）

- 书中提出的可操作步骤、检查清单、决策规则
- 可执行标准：能转化为具体步骤（不只是抽象原则）
- 可验证标准：有书中案例或实验数据支撑
- 可迁移标准：能应用于书中未提及的新场景

#### 2.3 核心概念词汇表（5-15个）

- 作者自创的术语
- 作者重新定义的常见术语
- 在书中结构性重要的术语

#### 2.4 案例模式（3-5个）

- 作者反复使用的论证模式/案例类型
- 每个模式：模式名称、典型案例、说明了什么

#### 2.5 应用范围与边界

- 这本书适合解决什么问题？
- 不适适合解决什么问题？
- 作者的盲区在哪里？

#### 2.6 表达风格（可选）

- 仅对风格明显的书籍提取
- 对教科书/技术书可跳过

### Phase 2.5: 提炼确认检查点

Phase 2 完成后，展示提炼摘要：

```
提炼结果摘要：
- 心智模型：N个（列出名称）
- 方法论/检查清单：N条
- 关键概念：N个
- 案例模式：N个
- 表达风格：[已提取/跳过]
```

用户确认OK → 进入Phase 3。
用户觉得不对或缺少 → 回到Phase 2调整。

---

### Phase 3: Skill构建

将Phase 2结果组装为可运行的SKILL.md。

#### Step 1: 读取模板
读取 `references/book-template.md` 获取标准结构。

#### Step 2: 填充内容

| 模板Section | 填充来源 |
|------------|---------|
| frontmatter description | 来源数量+模型数量+触发词 |
| 使用说明 | Phase 2.5 应用范围 |
| 回答工作流 | 根据心智模型自动推导研究维度 |
| 核心心智模型 | Phase 2.1 提取结果 |
| 方法论与检查清单 | Phase 2.2 提取结果 |
| 关键概念词汇表 | Phase 2.3 提取结果 |
| 案例模式 | Phase 2.4 提取结果 |
| 应用范围 | Phase 2.5 结果 |
| 书籍元信息 | Phase 1 书籍概览 |
| 诚实边界 | Phase 1 + Phase 2.5 结果 |
| 附录 | 章节分析来源 |

**回答工作流生成**：
书籍Skill不需要「先研究再回答」（框架本身就来自完整阅读），但需要：
1. 判断问题是否落入本书应用范围
2. 从心智模型和方法论中选择最相关的2-3个框架
3. 用选定框架分析问题，引用书中概念和案例

#### Step 3: 输出
将完成的SKILL.md写入 `.claude/skills/[book-slug]/SKILL.md`。

---

### Phase 4: 质量验证

#### 4.1 框架完整性测试
选3个书中明确讨论过的问题，用新Skill分析，结果应与书中结论一致。

#### 4.2 迁移测试
选1个书中未讨论但属于应用范围内的问题，用Skill的方法论框架推导出合理分析。
期望结果：基于框架X和方法论Y的推断，可能...但不确定。

#### 4.3 风格测试（如适用）
- 风格明显的书籍：输出是否保留了书的表达特征？
- 教科书/方法论书：输出是否清晰、结构化、可操作？

#### 4.4 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 心智模型数量 | 3-7个，每有来源证据 | <3或>10 |
| 方法论/清单数量 | 3-8条，可操作 | <3或全是泛泛原则 |
| 概念词汇表 | 5-15个，有定义 | 只列术语不给定义 |
| 应用边界 | 明确写适用范围和局限 | 只写优点不写局限 |
| 案例模式 | 至少3个，有书中案例支撑 | 无案例或案例不准确 |
| 章节覆盖 | 核心章节都被分析到 | 遗漏了高信息密度章节 |
| 表达风格（如适用） | 保留了书的风格特征 | 像通用AI味 |

**迭代上限**：Phase 2→4最多循环2次。如果2轮后仍有不通过项，在诚实边界中标注薄弱维度，交付当前最优版本。

**展示验证结果给用户确认后才算完成。**

## 更新已有Skill

当用户说「更新《XX》的skill」「这本书出了新版」时：
1. 读取现有SKILL.md，找到「书籍分析时间：[日期]」
2. 确认新版与旧版的差异（章节变化、概念增减、案例更新）
3. 只更新变化的部分，不重写整个Skill
4. 更新书籍分析时间和版次信息

---

## 品味守则

| 原则 | 一句话 |
|------|--------|
| 框架 > 摘要 | 书中反复出现的核心论点比章节摘要更有价值 |
| 方法论 > 感悟 | 可操作的步骤比「作者认为...很重要」更有用 |
| 边界 > 优点 | 知道这书不能干什么，比知道它能干什么更重要 |

### 绝不做的事
- 把通用道理包装成书的「独特见解」
- 忽略作者自己承认的局限
- 在信息不足时强行生成完整框架
- 把书的观点当成该领域的唯一真理

## 特殊场景

### 书籍类型差异

| 书籍类型 | 心智模型 | 方法论 | 概念表 | 案例模式 | 表达风格 |
|---------|---------|--------|--------|---------|---------|
| 哲学/理论 | 5-7个 | 1-3个 | 10-15个 | 3-5个 | 提取 |
| 方法论/实用 | 3-5个 | 5-8个 | 5-8个 | 2-3个 | 可选 |
| 叙事/传记/合集 | 3-5个 | 3-5个 | 5-10个 | 3-5个 | 提取 |
| 教科书/技术 | 2-4个 | 5-8个 | 10-20个 | 1-2个 | 跳过 |

### 信息不足的书
当搜索模式下发现公开信息很少时：
1. 告知用户「这本书的公开信息有限，生成质量会受限」
2. 心智模型减至2-3个，每个标注「基于有限信息推测」
3. 诚实边界加大篇幅

### 系列书籍
当用户说「蒸馏《XX》系列」时：
1. 确认是逐本蒸馏还是合并蒸馏
2. 合并蒸馏时：提取贯穿系列的核心框架 + 每本书的独特贡献
3. 在诚实边界中说明「系列合并可能丢失单本书的独特视角」

---

## 最后

一本书的价值不在于它说了什么，而在于它**怎么想**。

书骨抽走的不是摘要，是思维方式。读完就忘是正常——好的读者不需要记住每一页，只需要记住框架。
