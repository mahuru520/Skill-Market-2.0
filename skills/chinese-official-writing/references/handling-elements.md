# 办理要素

本文件用于起草前核对材料是否足够。它列的是办理要素，不是可直接复制的模板。

## 通用要素

多数公文或正式工作材料起草前，应尽量明确：

- 发文、报告、申请或项目主体。
- 主送、受文、阅读或执行对象。
- 需要办理、说明、请批、答复或协调的事项。
- 政策依据、会议依据、工作背景或现实原因。
- 当前情况、主要问题、风险或差距。
- 请求事项、工作要求、答复意见或后续安排。
- 完成期限、实施进度或反馈时限。
- 责任单位、牵头单位、配合单位或联系人。
- 附件、支撑材料、数据表或证明材料。
- 反馈渠道、报送方式或联系方式。
- 数据来源，以及数据是实际、测算、估算还是待补充。

可选要素缺失时，不要停止写作；关键审批事项、金额、期限、数据来源或主体关系缺失时，应提示用户补充。交付正文不得残留 `〔签发日期〕`、`〔会议时间〕`、`〔待补充〕`、`[具体项目名称]`、`XXXX万元`、`YYYY年MM月DD日`、`（签发日期）` 等未完成占位，除非用户明确要求输出占位模板。

## 文种要素表

| 文种/材料 | 必备要素 | 可选要素 | 常见问题 |
| --- | --- | --- | --- |
| 通知 | 对象、事项、要求、时限、材料或动作、反馈方式 | 依据、附件、逾期处理 | 只有“高度重视”，没有具体办理要求 |
| 请示 | 请示主体、一项请批事项、依据、必要性、方案或金额、风险控制、请批语 | 备选方案、附件、实施进度 | 多事并请、结尾像报告 |
| 报告 | 报告主体、背景、事实或进展、结果、问题、下一步 | 数据附件、责任单位、无需审批的协调建议 | 夹带“请批准”或结论埋得太深 |
| 决议 | 会议主体、审议事项、通过结果、执行要求 | 背景、责任分工 | 写成会议纪要或普通部署 |
| 说明 | 说明事项、原因、事实、影响、整改或后续安排 | 时间线、支撑附件 | 写成写作指南或政策宣讲 |
| 方案 | 目标、范围、任务、实施路径、进度、责任、保障、验收 | 预算、风险应对、数据管理 | 只有口号和概念，没有责任和时限 |
| 申请 | 申请主体、申请事项、依据、用途、金额或资源、使用计划、承诺 | 支撑附件 | 申请事项不清或缺使用计划 |
| 函 | 发函主体、受函单位、商洽事项、请求反馈或动作、期限、联系人 | 背景、附件 | 语气过于命令或过于随意 |
| 复函 | 来函依据、答复立场、办理要求、后续渠道 | 解释说明、附件 | 没有回应原问题 |
| 公示 | 发布主体、公示事项、依据、公示期限、异议渠道、联系方式、处理规则 | 附件名单、监督电话 | 缺公示期限、异议渠道或处理规则 |
| 征求意见函 | 发函主体、征求对象、事项背景、意见稿或条款、反馈期限、反馈方式、联系人 | 附件、邮件标题要求 | 只说征求意见，没有反馈路径 |
| 采购公告 | 采购主体、项目名称、预算或上限、采购内容、资格要求、响应期限、提交方式、联系人 | 评审办法、附件、合同边界 | 缺预算、期限、提交方式或联系方式 |
| 批复 | 来文依据、批复决定、实施要求、监督或报送要求 | 条件、经费控制、验收 | 没有明确同意、不同意或调整决定 |
| 决定 | 决定事项、依据、适用范围、执行要求、生效时间 | 责任分工、解释机关 | 权限来源不清 |
| 意见 | 发文主体、重要问题、总体要求、政策措施、责任机制 | 工作目标、保障措施 | 只有原则，没有措施、场景或责任 |
| 命令（令） | 发布主体、命令事项、依据、生效或执行要求 | 附件、编号 | 用于普通工作部署 |
| 公报 | 发布主体、公开事项、权威数据或决定、发布时间 | 背景说明 | 写入不宜公开的内部流程 |
| 公告 | 发布主体、公开对象、公告事项、时间地点或办理方式、责任提醒 | 依据、联系方式、附件 | 写成内部通知或夹带内部流转 |
| 通告 | 发布主体、适用范围、应遵守或周知事项、管理要求、生效时间 | 依据、监督方式、责任提醒 | 写成泛泛公告或内部通知 |
| 议案 | 提请主体、审议机关、议案事项、依据、理由、拟议内容 | 附件、说明材料 | 把普通建议写成议案 |
| 通报 | 情况、发现、评价、问题或表扬、要求、整改动作 | 典型案例、期限 | 写成通知或宣传稿 |
| 会议纪要 | 会议时间、主题、参会单位、议定事项、责任、期限 | 未决事项、下次会议 | 写成会议新闻 |
| 工作要点 | 年度或阶段目标、重点任务、责任机制、时间节点、评价方式 | 支撑项目、风险控制 | 任务清单没有重点和措施 |
| 工作总结 | 已完成工作、数据或成效、经验、问题、下一步 | 典型案例、附件 | 表扬过多，事实不足 |
| 调研报告 | 主题、方法、事实、发现、问题、原因、建议 | 样本范围、数据来源 | 建议没有调研事实支撑 |
| 可研报告 | 需求、必要性、可行性、方案比较、成本、风险、实施、成效 | 政策依据、融资、进度 | 结论没有测算或条件 |
| 实施方案 | 目标、阶段、任务、责任、进度、保障、验收 | 宣传培训、数据流转 | 没有可追责主体 |
| 建设方案 | 背景、需求、建设内容、资源和成本、运维、成效、安全 | 架构、采购方式、推广安排 | 必要性和建设内容反复重复 |
| 审查材料 | 审查对象、依据、发现、风险、整改要求、结论 | 证据清单、后续复查 | 只写“基本合规”，没有条件和问题 |

## AI 算力和技术服务要素

仅在任务明确涉及算力、GPU/服务器租赁、云端部署和模型服务时核对本节；通用公文和普通事务材料跳过。普通采购公告、费用申请和办公设备申请不默认进入 AI 算力语境。

先核对需求来源、资源或 Token 测算、成本边界、SLA/并发/安全、交付和验收是否存在；详细测算和参数转读 `ai-compute-docs.md`，不要在通用材料中展开技术参数清单。

## 缺项处理

- 不编造真实单位、金额、政策、日期、电话、邮箱、文号、签发人和印章。
- 默认不外搜补缺项。用户明确要求搜索或核验公开来源，或任务含“最新”“当前”“今日”“现行政策”“近期数据”等时效事实时，才可联网核验；搜索结果只作事实核查和来源参考。来源冲突、无法核验或工具不可用时，列为待确认事项，不写成已确认事实。
- 不因出现单位名称就搜索单位公开样文、固定格式或写作风格；用户未给单位模板时，使用中性称谓和通用文种格式。
- 缺项优先在正文外用修订说明提示；不要把 `〔签发日期〕`、`〔会议时间〕`、`〔维护开始时间〕`、`[具体项目名称]`、`XXXX万元`、`YYYY年MM月DD日`、`（签发日期）`、`根据用户提供材料填写` 等未完成字段留在交付正文中。
- 用户要求完整落款但未说明成文日期缺失或待确认时，可使用当前日期作为草稿日期；用户明示成文日期缺失、待确认或需另行确认时，不使用当前日期补落款，只在正文外列待确认。正式签发日期、文号、印章和签发人仍不得编造。
- 当前日期只可用于草稿落款，不可替代会议时间、维护时间、实施期限、政策依据或业务数据；这类关键缺项应在正文外提示用户确认，或改写为不含具体日期/数值的稳妥表述。
- 如果缺项影响文种成立，例如请示没有请批事项、批复没有来文依据，应优先补问或标注风险。请示、报告、上报申请缺主送机关、发文或申请单位、成文日期时，不编造“上级单位”“申请单位”等泛称；先完成内容草稿，并把这些结构缺口列在待确认事项最前，标明影响正式报送和文种完整性。
