# 公文语言风格

正式文稿应清楚、具体、面向办理和决策。读者应能直接看出事项、依据、判断和下一步要求，而不是被讲授概念或写作方法。

## 主体视角

起草前先确定说话位置：

- 发文机关或主管单位视角：强调政策依据、管理要求、组织实施和监督落实。
- 报告单位视角：强调已做工作、客观情况、问题风险和下一步安排。
- 项目单位视角：强调实施条件、服务范围、进度、运维、验收和责任。
- 企业或集团视角：强调经营价值、成本控制、资源统筹和风险管理。
- 调研或审查视角：强调事实来源、分析判断、审查意见和建议。

全文保持同一视角。除非用户底稿已有明确风格，不要在“我单位”“本项目”“公司”“外部观察者”之间来回切换。

## 段落形态

把以下顺序作为写作习惯，不写成可见标签：

1. 主要判断或办理事项。
2. 事实、依据或数据支撑。
3. 实际影响、问题判断或必要性。
4. 项目安排、责任要求或后续落点。

示例：

`经测算，现有业务在全面接入后将形成较高的年度 Token 调用需求。智能审校系统、内容生产平台、行业知识库和编辑审校工作流持续扩大使用范围，长文本处理、多轮修改和批量任务会同步增加调用量。项目通过租赁方式集中取得算力服务，有利于稳定支撑未来三年业务放量。`

## 常用表达

按语境选用，不机械套用：

- `围绕……需求，项目拟采用……方式实施。`
- `经测算，……年度需求约为……。`
- `从业务运行看，……已形成持续调用需求。`
- `从成本变化看，……将推高后续综合费用。`
- `从安全管理看，……需要在受控环境内处理。`
- `项目实施后，将形成……能力。`
- `该方式将……纳入服务合同管理，便于按服务期限、服务范围和运行效果开展履约管理。`
- `在优先保障核心业务的前提下，可结合资源余量开展……。`
- `相关测算将在实际日志和结算数据形成后动态校准。`

## 用词偏好

- 优先使用具体业务名词，少用“赋能、底座、生态、抓手”等抽象词。
- 评价强度与证据强度匹配。使用“提升、强化、推进、赋能、闭环、机制、体系、显著、长效”等抽象词或强评价时，后面应落到具体对象、动作、责任、时限、数据或制度载体；没有证据时改为审慎表述或列入待确认。
- 涉及费用时，用“费用边界”“服务周期”“履约管理”“成本可控性”等能支撑决策的表达。
- 涉及测算时，用“经测算”“按现有数据测算”“结合业务增长测算”，并说明数据性质。
- 涉及判断时，用“可形成”“有利于”“将支撑”“能够降低”等审慎结论。
- 涉及安全时，用“受控环境”“权限可管”“日志可审”“调用可追溯”等可检查表达。

## 行文语气

- 上行文：克制、准确、有依据；避免替上级作结论。
- 下行文：明确、可执行、有时限；避免只喊重视。
- 平行文：平等、具体、留有协商空间；避免命令式语气。
- 内部材料：可比正式发文略紧凑，但仍应有事实、责任和落点。

## 轻量语气替换建议

以下只作建议层，不新增硬清洗，不把真实基层口吻、引述内容或用户明确保留的表达一律清除。去口语化、顺稿和审稿时必须保留原文事实，不新增原文没有的依据、数据、成效或责任安排。

- `我觉得`：材料只表达初步意见时，可改为 `初步考虑` 或 `初步判断`；只有材料确有研究过程或事实依据时，才使用 `经研究认为` 或 `从实际情况看`。
- `搞`：可改为 `开展`、`推进`、`实施`、`组织`。
- `差不多`：可改为 `基本具备条件`、`总体可行`、`需进一步核实`。
- `马上`：可改为 `立即`、`尽快`、`按时限推进`。
- `然后`：可改为 `随后`、`同步`、`并`、`在此基础上`。

如果用户只要求语气检查，优先给出位置、风险层级和修改建议；只有用户要求代改时才输出改后正文。

去 AI 味或语气审稿应匹配文体。公文和正式工作材料以克制、准确、可执行为“自然”，不为了显得像人写而加入第一人称、口语化、情绪化评价、玩笑或散文化表达。检查时看成簇问题：抽象词、强评价、机械结构、模糊归因和缺少办理落点同时出现时再提示压实；单个正式词或单个转折不作为硬清洗理由。

正式化改写只压实原文已有事实。不要为了让句子更正式而补入原文没有的组织名称、牵头部门、责任分工、整改动作、督办安排、成果总结或后续处理进展；这些信息缺失时，在正文外提示补充。

口语来源不等于事实授权。`老板关心`、`钱花得值`、`马上要搞` 只能说明用户材料中出现了关注、费用判断或推进意向，不得自动升级成 `领导高度关注`、`投入产出清晰`、`推进较为紧迫`、`按程序推进` 等强判断；如需正式化，应使用低强度表达，并把程序、结论、时限和审批态度留给用户确认。

## 应避免的写法

- 不写 `本方案重点说明三个问题`、`本文将从以下方面展开` 等旁白句。
- 不教读者如何理解概念或如何写报告。
- 不把技术材料写成概念科普，除非读者确实需要定义关键术语。
- 不把正式文稿写成口语，如将 `更稳，也更省` 改为 `成本和服务保障更具确定性`。
- 不堆砌“全面推进、持续深化、不断提升、形成闭环”等没有对象、机制或结果的套话。
- 强判断须有复查记录、运行数据、制度文件或责任闭环支撑；证据只覆盖局部时，结论范围和强度随之收窄。
- 不把字段卡片直接当成正文，除非用户要求表格式摘要。
