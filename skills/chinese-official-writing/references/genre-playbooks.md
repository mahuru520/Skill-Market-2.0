# 文种与专项 Playbook

本文件用于命中具体文种或专项时快速进入正确骨架。每个文种小节都是可从 `SKILL.md` 直接进入的叶子路由，只给适用边界、输入要素、结构骨架、常见误区和条件式补充读取。

## 目录

- 使用方式
- 会议纪要
- 请示/申请
- 报告/情况说明
- 函/复函/征求意见函
- 通知/通告/公告/公示/通报
- 意见/决定/决议/议案/公报/命令
- 讲话稿/致辞/述职

- 调研报告/研究报告/可研报告/建设方案
- 采购公告/审查材料
- AI 算力与技术服务

## 使用方式

- 从 `SKILL.md` 直接进入命中的文种或专项小节，不要求先完整读取 `workflow.md` 或 `genre-routing.md`。只有任务同时命中复杂改稿、多材料合稿、文种/行文关系不明等条件时，才补读对应长 reference；不要把每节末尾的“补充读取”当成固定加载清单。
- 每节只用于确定材料骨架和风险点。用户已有模板和字段顺序优先，不因 playbook 改掉真实模板、主送、落款、字段或附件关系。
- 字段式、表单式、清单式材料先保留原字段换行、字段名、字段顺序和非目标字段值；用户只要求修改某一字段时，只替换该字段内容，不把多字段合并成一句。分号只是字段分隔符，拆成独立字段行后不要保留行尾分号或造成 `。；`。
- 事实不足仍按正文后短列待确认处理，不新增默认联网、API、Word/PDF 或脚本硬门禁。正文外待确认、风险提醒、来源核验或引用核验提示不算文种主体章节，不接续正文编号。
## 会议纪要

- 适用：会议记录、录音整理、议题讨论稿需要转成纪要；重点是议定事项、责任、期限和后续动作。
- 不适用：会议新闻、宣传报道、个人笔记、逐字稿全文整理。
- 骨架：会议主题和范围 -> 议题或讨论要点 -> 议定事项 -> 责任单位/责任人 -> 完成期限 -> 未决事项。
- 路径：决策会重“审议、原则同意、决定、要求”；办公会重“听取、研究、明确、要求”；专题会重“听取、研究、明确”；座谈会重“听取、认为、指出”。材料未给会议结论时，只整理事实和待确认项。同一句中的主体、动作、期限应保持绑定，不把有期限的事项拆成多个待明确字段，也不把一个期限扩展到原文未绑定的其他任务。
- 风险：不写成会议新闻；不把发言顺序当成结论顺序；未给结论时标为待确认，不编造“会议决定”；原材料只给议题、责任和期限时，不补写“会议认为”“会议强调”等未给会议判断，也不补写问题类型、产生环节、处理建议、保障效果等未给交付物；责任或期限未给时留空或列为待确认，不用“按审核执行”“后续推进”等泛口径补期限。
- 补充读取：`genre-checklist.md` 的纪要小节、`review-checklist.md` 的只审不改规则。

## 请示/申请

- 适用：向上级、领导班子或审批方请求批准经费、事项、资源、立项、采购方式。
- 骨架：请批事项 -> 依据和必要性 -> 方案/金额/资源 -> 风险控制 -> 请批语。
- 风险：请示一文一事；申请可保留用户模板和称呼，不因出现“妥否，请批示”就强行改成请示。请示、报告和上报申请的主送机关、发文或申请单位、成文日期属于正式报送结构要素；未知时不编造泛称，也不在正文留未完成占位，应列为待确认事项前列并标注影响正式报送。
- 补充读取：`argument-chains.md` 的请示和请批附件、`formal-addressing.md`。

## 报告/情况说明

- 适用：报告进展、情况、结果、问题、整改或说明事实。
- 骨架：报告事项和范围 -> 已有事实/进展 -> 问题或原因 -> 已给影响、处置或建议；材料未给影响、处置和安排时，不为补齐骨架硬写，改用正文外待补事项。
- 使用/体验/评估报告或成本考察：用户说“使用报告、体验报告、服务使用情况、工具使用情况、评估报告、成本考察、成本评估”时，不自动改题为“调研报告”“考核说明”或“实施方案”。骨架可用“基本使用情况 -> 使用成效 -> 存在问题 -> 影响分析 -> 优化建议 -> 后续使用建议/总体结论”；问题要充分写清，但不要让负面内容吞没准确性、可用价值、适用场景和后续观察安排。材料只给“建议尝试、考虑测试、下一步设想”时，写成优化建议或测试安排建议，不写成已经确定的执行路线、责任命令或反馈时限。
- 风险：报告不写审批请求；情况说明重点交代已给原因、影响、处置，不替上级作决定。待确认事项、风险提醒或引用核验提示放在正文外，不编号成报告的最后一章。用户写明“材料只有”或事实很少时，宁可短写，不把“完成梳理、组织协调、发现问题”扩展成系统范围、字段对象、台账清单、讨论内容、对接要求、责任链条、督办闭环、可验收结论等未给事实；材料只说接口、系统、页面异常时，按用户原词写接口、系统或页面，不包装成核心系统、生产系统、业务主系统，也不补原因结论、责任部门、损失金额或整改方案。
- 补充读取：`argument-chains.md` 的报告和总结、`review-checklist.md` 的事实强判断。

## 函/复函/征求意见函

- 适用：平行商洽、答复来文、征求意见或协调事项。
- 骨架：来文依据或事项背景 -> 商请/答复/征求事项 -> 办理意见 -> 反馈期限和渠道 -> 联系人和附件。
- 风险：复函重回应来文；函不写成命令；征求意见函必须给反馈路径。
- 补充读取：`formal-addressing.md`、`genre-checklist.md` 的函和征求意见函小节。

## 通知/通告/公告/公示/通报

- 适用：布置办理、公开周知、公示名单或事项、通报已知情况和明确要求。
- 骨架：事项 -> 对象范围或已知事实 -> 已给问题或要求 -> 已给时限/渠道/联系人；材料未给执行要求、整改要求和后续处理时，不为了像通报而补写。
- 风险：通知重执行，公告/通告重公开范围，公示重异议渠道，通报重事实和已给要求；不要互相套用。通报只给“已完成梳理、发现问题、组织协调会、后续需补充”时，可用短骨架“已完成事项 -> 发现问题 -> 已组织协调（只陈述次数或事实） -> 补充以下信息后，文章会更完整”；协调会只写“已组织2次协调会”，不展开会议内容、处理边界、问题清单、对接要求、闭环机制或督办安排，也不写“对关键问题讨论、沟通、归集、说明、形成意见”；材料未给整改、处置、下一步安排时，不新增“已开展处置”“下一步安排”“整改方向”“治理闭环”“督导流程”等章节或小标题。用户第二轮指出补造事实或要求删未给内容时，按 `workflow.md` 的事实映射式二次修改删掉未支持推断；待补责任人、更新周期等放正文外或写成待补口径，按用户原词短列，不加括号解释、对象范围、岗位、联系方式、生效区间、牵头说明或补齐时限；必要衔接不写“以便推进/落地/衔接”等目的判断。
- 补充读取：`genre-routing.md` 和 `handling-elements.md` 的对应行。

## 意见/决定/决议/议案/公报/命令

- 适用：规范性部署、会议审议结果、提请审议、权威发布或命令事项。
- 骨架：权限和依据 -> 事项或决定 -> 适用范围 -> 执行要求 -> 生效或后续程序。
- 风险：命令（令）不用于普通内部部署；公报只写适合公开的权威事项；议案不能写成一般建议。
- 补充读取：`genre-checklist.md` 的对应法定文种小节。

## 讲话稿/致辞/述职

- 适用：正式场合发言、活动致辞、述职报告等事务材料。
- 骨架：场合身份 -> 主题态度 -> 事实或工作基础 -> 重点任务/体会 -> 收束到责任或目标。
- 风险：可比公文更有感染力，但不能空泛排比、编造成效或写成宣传口号堆叠；称呼和受众按材料给定，未给同学、领导、嘉宾等对象时，不自行补受众称呼。
- 补充读取：`official-style.md` 和 `anti-ai-patterns.md`。

## 工作总结/工作要点/周报

- 适用：阶段总结、年度要点、周报或月报。
- 骨架：总结写已完成、成效、问题、经验、下一步；工作要点写目标、重点任务、责任机制、时间节点、评价方式；周报写已完成、推进中、问题、下步。
- 风险：不把所有总结改成周报格式；字段式周报保留字段和换行，不散文化、不合并字段；责任机制只写材料给出的主体，未给运维服务单位或合同责任时，不补服务单位责任；成效必须有事实支撑。
- 补充读取：`argument-chains.md` 的报告和总结。

## 调研报告/研究报告/可研报告/建设方案

- 适用：专题调研、研究分析、可行性论证、建设方案。
- 骨架：对象和范围 -> 事实、数据、样本 -> 发现和问题 -> 原因或方案比较 -> 建议/可行性/建设内容 -> 条件和风险。
- 路径：调研报告先锁定对象、样本、事实和问题；可研报告先核对项目类型、建设性质、建设内容、投资估算、资金来源、实施条件、风险和效益；建设方案先核对目标、范围、任务、进度、责任和验收。
- 风险：不把公文报告写成投研报告；建议必须回到调研事实；可研结论要说明条件和测算边界。项目类型、资金来源、投资估算或测算口径缺失时列正文外待确认，不用地方模板或空表补实。
- 补充读取：`argument-chains.md` 的可研、调研和审查，专项技术材料再读 `ai-compute-docs.md`。

## 采购公告/审查材料

- 适用：公开采购事项、审查意见、初步设计或项目材料审查。
- 骨架：采购公告写采购主体、项目名称、预算、内容、资格、响应期限、提交方式、联系人；审查材料重点是依据、发现、风险、整改要求和结论。
- 路径：采购申请、采购公告和采购审查优先核对品名、规格、数量、预算或单价、供应商/报价依据、提交方式、验收和归档要求；已给的信息写入正文，未给的供应商、报价依据、验收安排或归档要求列正文外待补，不写成正文里的流程承诺。采购需求不等于现有能力不足、效率提升、业务范围不变或影响结论；用户只给预算时写“预算为”，不包装成“经测算”。
- 风险：普通采购公告不默认进入 AI 算力语境；字段式审查材料只改用户指定字段，其他字段按原行、原字段名和原值保留，拆行时不保留原分号为行尾噪声；不得补造品牌、型号、供应商或评审结论。
- 补充读取：`handling-elements.md` 和 `genre-checklist.md` 的采购公告、审查材料小节。

## AI 算力与技术服务

- 适用：明确涉及 AI 算力、GPU/服务器租赁、模型服务、智算中心、云端/本地部署比较、SLA、安全或验收的正式材料。
- 骨架：需求来源 -> 资源或 Token 测算 -> 成本边界 -> 部署/租赁/采购路径 -> SLA/并发/安全 -> 交付和验收。
- 风险：不要把普通采购、费用申请或办公设备申请写成算力论证；不要写“先进算力”“强大平台”等无指标判断。
- 详细结构转读 `references/ai-compute-docs.md`；本文件只保留触发和边界。
