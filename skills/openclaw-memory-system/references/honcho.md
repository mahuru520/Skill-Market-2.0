# Honcho 用户画像

## 概念

从对话和行为中隐式提取用户特征，用于优化回复策略。不是显式问卷，而是"通过观察学习"。

六大维度：

| 维度 | 含义 | 取值 |
|------|------|------|
| 技术水平 (Skill Level) | 对技术概念的理解深度 | 0.0~1.0 |
| 交互节奏 (Pacing) | 对话推进速度 | 0.0~1.0 |
| 沟通风格 (Style) | 正式度/简洁度/幽默感 | 多维 0.0~1.0 |
| 目标取向 (Goal Orientation) | 完成型 vs 探索型 | 0.0~1.0 |
| 情绪基调 (Emotional Tone) | 情绪稳定性/压力反应 | 定性 |
| 矛盾偏好 (Contradictory Preference) | 内在矛盾强度 | 0.0~1.0 |

## 画像文件

`HONCHO.md` 放在 workspace 根目录，每次 heartbeat 迭代更新。

## 注入策略（使用画像）

画像不是给用户看的，是给 AI 自己参考的。用法：

### 回复长度
- 技术调试/深度研究：800-1500 字，可长
- 查询/操作类：200-400 字，简洁
- 不确定时：先给结论 + "要深挖吗？"

### 句式风格
- 简短直接，无寒暄
- 技术术语精确使用
- 不说"当然可以"、"我来帮你"等废话

### 矛盾场景应对
例如用户问原理又问速度：
- 先给结论 + 关键原理一句话
- 然后问"要细看哪部分？"

## 迭代机制

HEARTBEAT.md 中配置迭代任务：

```markdown
### Honcho 画像更新（每 2~3 天一次）
从最近 memory/*.md 中提取对话事件，更新 HONCHO.md：
- 新项目/任务 → 技术水平、目标取向
- 对话风格变化 → 沟通风格
- 矛盾信号出现 → 矛盾偏好强度
- 长对话或情绪异常 → 情绪基调
```

## 建立画像的对话信号

| 信号 | 推断维度 |
|------|---------|
| 直接问"怎么配置"开始操作 | 技术水平 0.7+，节奏快 |
| 改源码、debug MCP | 技术水平 0.75+ |
| 说"不要寒暄"、"直接说" | 简洁度 0.75+ |
| 问完原理又问实现 | 矛盾偏好 0.7+，探索型 |
| 卡点不抱怨，直接排查 | 情绪稳定 |
| 操作完整理成技能 | 完成型 + 沉淀意愿 |
| 新闻速查后跟深挖 | 感兴趣点才慢，其他快 |

## 画像时间线

每次重大事件后更新画像，更新时在 HONCHO.md 底部的更新日志记录：
- 时间
- 事件
- 维度变化

## 核心矛盾的应用

`矛盾偏好（想要快速出结果，又想理解底层原理）` 是 pooki 的核心矛盾。

注入策略：
1. 先结论（满足"快"的需求）
2. 一句话原理（满足"理解"的需求）
3. 主动问"要细看哪部分？"（让用户选，不强塞信息）

```
用户问 → 先结论 + 1句原理 + "要深挖吗？"
```
